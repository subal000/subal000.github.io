How we handled 10k events per second in Flink without crashing our DB

When you are processing thousands of financial events every second from Kafka, you quickly realize the biggest enemy is not the math, it is network latency.

To get useful data out of our stream, we had to do three things for every single transaction in real time:

  1. Calculate the USD value using a volatile price.
  2. Map the fromToken to a human-readable name.
  3. Map the toToken to a human-readable name.

The easy way is just writing a map function that hits an API or Redis for every record. That works fine on your laptop with 5 events a second. But when you hit 10k events a second in production, your database connections just give up, you burn through your API limits, and your Kafka lag shoots through the roof.

We fixed this by using Flink’s built-in stateful features to make a self-sustaining pipeline. Here is how we got rid of the external network calls and handled complex, multi-key joins right in the cluster memory.

The setup: DEX swaps

To give you an idea of the data we were dealing with, we were processing real-time DEX swap data from the Solana blockchain, flowing out of a Kafka topic.

A raw payload looks like this:

{
  "signature": "3ucNos4Nbum...",
  "from_mint": "So11111111111111111111111111111111111111112",
  "to_mint": "EPjFWdd5AufqSSqeM2qN1xzybapC8G4wEGGkZwyTDt1v",
  "from_amount": 15.5,
  "to_amount": 2100.25
}

This payload is small and fast. But to make it useful for analytics in StarRocks, we need those token symbols (turning So1111... into SOL) and the dollar value. Since new tokens pop up all the time, we couldnt just hardcode them. We had to build something dynamic that didnt rely on blocking network calls.

1. Building our own in-stream price oracle

Instead of reaching out to a price API for every event, we realized the market price was already inside our stream because of the high liquidity trading pools.

We used a two step pattern: Extract and Broadcast.

We skimmed high volume pools using a flat-mapper (SolPriceExtractor). Whenever a trade between our main asset and a stablecoin happened, we calculated the price and threw it into a pricing stream. Then we used Flink's BroadcastStream to push that price to every parallel worker in the cluster, keeping the state local to the execution memory.

// Skim the price out of the raw traffic and broadcast it
BroadcastStream<PoolPriceData> broadcastPriceStream = exchangeStream
        .flatMap(new SolPriceExtractor())
        .broadcast(priceStateDescriptor);

// Connect it back to the main processing engine
DataStream<SwapWithPrice> streamWithSolPrice = exchangeStream
        .connect(broadcastPriceStream)
        .process(new UsdValueCalculator());

To keep things smooth, our UsdValueCalculator maintains a rolling list of the last 20 prices in memory. By averaging this, we get zero-latency pricing. If the pipeline just started and the state was empty, we had a simple defensive fallback that hit an API with a 60 second cache, just enough to keep the stream moving until the local broadcast state caught up.

2. Solving the multi-key join paradox

Once we had the pricing, we hit a Flink bottleneck. Each swap event needs metadata for two distinct tokens (fromToken and toToken).

The problem is that in Flink, you can only keyBy on one key at a time. You cant route a single record to two places at once without causing massive network shuffles or data duplication. We fixed this by turning the enrichment into a two-stage assembly line.

  • Stage 1: We partition the stream by fromToken and route it to EnrichFromSymbolFunction. It checks local RocksDB state (pre-loaded from our analytics warehouse). If it hits, it appends the symbol. If not, it hits an API and saves it for next time.
  • The handoff: We wrap the result in a temporary SwapWithFromSymbol POJO. Then, we force the stream to reshuffle across the network, this time using the toToken as our routing key.
  • Stage 2: The stream hits EnrichToSymbolFunction. It performs the same RocksDB lookup for the second token.

By breaking a multi-dimensional join into this pipelined sequence, we kept everything in memory and avoided the massive cost of hitting a database twice for every single row.

3. Dynamic topology

Finally, we wanted to be able to change where our data lands without rewriting our code. We added a simple parameter SINK_TYPE that lets us decide where the data goes when we deploy.

String sinkType = params.get("SINK_TYPE", "kafka");

if ("kafka".equalsIgnoreCase(sinkType)) {
    enrichedStream.sinkTo(createKafkaSink(params));
} else if ("starrocks".equalsIgnoreCase(sinkType)) {
    enrichedStream.addSink(createStarRocksSink(params));
}

This let us test locally against Kafka and then just flip a switch to send it to our analytics warehouse in production.

What we learned

By ignoring the urge to just throw Redis at the problem, we kept our data gravity contained within our compute nodes. If you are dealing with high-throughput streams, just remember:

  • Listen to your own noise: Use Broadcast State to extract metadata hidden inside your primary stream.
  • Reshape mid-flight: If you have a multi-key join, just split it into an assembly line of sequential shuffles.
  • Parameterize your topology: Decouple your logic from the infrastructure so you arent locked into one storage layer.

It makes the whole pipeline way easier to manage once you stop relying on external stuff for every little thing.
