How we handled 10k events per second in Flink without crashing our DB

When you are processing thousands of financial events every second from Kafka, the biggest enemy is not the math. It is network latency. And the worst part is that the naive solution works perfectly until it doesn't. Five events per second on your laptop: fine. Five hundred in staging: fine. Ten thousand in production on a Monday morning: your database connection pool is exhausted, your API rate limits are blown, and your Kafka consumer group lag is climbing faster than you can read it.

This is the story of how we fixed that for a real-time DEX swap enrichment pipeline on Solana.

What we were building, and why it is harder than it sounds

A decentralized exchange (DEX) swap is a transaction where a user trades one token for another directly on-chain, without a centralized intermediary. On Solana, these happen constantly -- thousands per second across dozens of liquidity pools. We were ingesting raw swap events from Kafka and needed to turn them into analytics-ready records in StarRocks before downstream dashboards could use them.

A raw event looks like this:

{
  "signature": "3ucNos4Nbum...",
  "from_mint": "So11111111111111111111111111111111111111112",
  "to_mint": "EPjFWdd5AufqSSqeM2qN1xzybapC8G4wEGGkZwyTDt1v",
  "from_amount": 15.5,
  "to_amount": 2100.25
}

That payload is compact and fast. But to make it useful for analytics, we needed three things for every single record:

  1. The USD value of the trade. Without this, you cannot aggregate volume or compare trades across different token pairs.
  2. A human-readable symbol for from_mint. So1111... means nothing to an analyst. SOL does.
  3. A human-readable symbol for to_mint. Same problem, different token address.

The obvious solution -- hitting a price API and a token metadata API for every record -- works until throughput climbs. Then it falls apart in three ways simultaneously: your API provider rate-limits you, your database connection pool runs out of connections, and Kafka lag accumulates because your pipeline is spending most of its time waiting on network I/O rather than processing records. We hit all three on the same day.

The fix required rethinking where state lives and how joins work inside Flink.

Pipeline overview

1. Building our own in-stream price oracle

The first instinct when you need a price feed is to call an external API -- DexScreener, CoinGecko, whatever. The problem is that at 10k events per second, even a cached API call with a 1-second TTL is still 10,000 cache reads per second. That is load you do not need to put on anything external.

The price was already inside our stream. Because high-liquidity pools like the SOL/USDC pair on Orca trade constantly, every few milliseconds there is a swap event that implicitly encodes the current market price. If someone swaps 15.5 SOL for 2100.25 USDC, the price of SOL at that moment is 2100.25 / 15.5 = ~$135.50. We were throwing that information away.

We fixed this with a two-step pattern: extract, then broadcast.

graph TD Raw[Raw Event Stream] --> |flatMap| Filter[Filter to SOL/USDC Pools] Filter --> |Extract| PoolData[PoolPriceData Stream] PoolData --> |broadcast| Bcast[BroadcastStream] Bcast --> |connect| Calc[UsdValueCalculator] Exchange[ExchangeStream] --> |connect| Calc Calc --> Out[SwapWithPrice Stream] style Bcast fill:#f9f,stroke:#333 style Calc fill:#4ecdc4,stroke:#333

First, we defined the pools we trusted as price sources -- high-volume, well-established SOL/USDC and SOL/USDT pools where the price signal is reliable:

final static Set<String> SOL_USDC_POOLS = new HashSet<>(Arrays.asList(
    "Czfq3xZZDmsdGdUyrNLtRhGc47cXcZtLG4crryfu44zE",
    "58oQChx4yWmvKdwLLZzBi4ChoCc2fqCUWBkwMihLYQo2",
    "FbkX1h2YTs171cEMa4GrV7XbAiQt5zSmV2CjfYWxXJDP",
    "HSTnyz57adepabCQyfmMST5eKfAsi6prAN1Ch2St8J7P"
));

Then we used a FlatMapFunction called SolPriceExtractor to skim price signals out of the main event stream. The price calculation handles both trade directions (SOL to USDC and USDC to SOL):

public static class SolPriceExtractor implements FlatMapFunction<ExchangeData, PoolPriceData> {
    @Override
    public void flatMap(ExchangeData exchange, Collector<PoolPriceData> out) {
        if (SOL_USDC_POOLS.contains(exchange.pair)) {
            double solPrice = 0.0;
            if (SOL_ADDRESS.equals(exchange.fromToken) && USDC_ADDRESS.equals(exchange.toToken)
                && exchange.fromAmount > 0) {
                solPrice = exchange.toAmount / exchange.fromAmount;
            } else if (USDC_ADDRESS.equals(exchange.fromToken) && SOL_ADDRESS.equals(exchange.toToken)
                && exchange.toAmount > 0) {
                solPrice = exchange.fromAmount / exchange.toAmount;
            }
            if (solPrice > 0) {
                out.collect(new PoolPriceData(exchange.timestamp, solPrice, exchange.pair));
            }
        }
    }
}

Then we used Flink's BroadcastStream to push that price to every parallel worker in the cluster:

BroadcastStream<PoolPriceData> broadcastPriceStream = exchangeStream
    .flatMap(new SolPriceExtractor())
    .broadcast(priceStateDescriptor);

DataStream<SwapWithPrice> streamWithSolPrice = exchangeStream
    .connect(broadcastPriceStream)
    .process(new UsdValueCalculator());

Inside UsdValueCalculator, we maintain a rolling window of the last 20 prices and use their average. Why 20? It balances responsiveness against noise. At our trading volumes, high-frequency wash trades can cause momentary price spikes that distort volume calculations. Averaging over 20 prices (which might span 200 to 500ms of actual trading) smooths those out without introducing meaningful lag for a metric that is already approximate. We considered a time-based window (the last 5 seconds for example), but that requires event-time watermarks and adds complexity for minimal gain -- 20 trades is simpler and robust enough at our volumes.

The fallback matters too. When the job first starts, the broadcast state is empty. Rather than outputting records with a zero price or dropping them, we have a lightweight API call with a 60-second cache to bridge the gap:

if (solPrice <= 0) {
    long currentTime = System.currentTimeMillis();
    if (cachedApiSolPrice > 0 && (currentTime - lastApiFetchTime) < CACHE_DURATION_MS) {
        solPrice = cachedApiSolPrice;
    } else {
        solPrice = fetchSolPriceFromApi();
        if (solPrice > 0) {
            cachedApiSolPrice = solPrice;
            lastApiFetchTime = currentTime;
        }
    }
}

In practice, the fallback fires maybe twice per job restart -- once on cold start, and once if the broadcast state somehow drains. After that, the stream self-sustains.

2. Solving the multi-key join problem

Once we had USD pricing flowing, we hit a subtler Flink constraint. Each swap has two token addresses that both need metadata enrichment: fromToken and toToken. A naive approach -- doing both lookups inside a single map function -- runs into the same external-call problem we just escaped. But the Flink-native approach has its own limitation: you can only keyBy on one key at a time.

This is a hard architectural constraint in Flink. Partitioning a stream by a key means routing all records with the same key to the same task slot, which is what makes keyed state coherent. If you try to keyBy on two keys at once, you would need to duplicate records (sending each swap to both the fromToken slot and the toToken slot), which doubles your data volume and creates complex join logic downstream.

Our solution was to turn the two-dimensional join into a sequential assembly line. Rather than trying to enrich both tokens at once, we enrich one at a time and carry the partial result forward.

Why RocksDB

Before getting into the code, the state backend choice is worth explaining. Flink supports two main backends: the JVM heap (HashMapStateBackend) and RocksDB (EmbeddedRocksDBStateBackend). For this pipeline, we used RocksDB because token metadata is unbounded -- Solana has hundreds of thousands of tokens, with new ones launching daily. Keeping all that in JVM heap would cause GC pressure and eventually OOM errors. RocksDB stores state on local disk with an in-memory cache, so our per-task state can grow well beyond what heap allows, while still serving the common case (already-seen tokens) from memory.

env.setStateBackend(new EmbeddedRocksDBStateBackend(true));

The true argument enables incremental checkpointing, which is important for a long-running job -- full state snapshots at our data volume would be prohibitively slow.

Pre-loading state from StarRocks

Rather than starting cold and hitting the fallback API for every token on day one, we pre-load known token metadata from our analytics warehouse at job startup. The Flink Table API makes this straightforward:

tEnv.executeSql(
    "CREATE TEMPORARY TABLE starrocks_token_metadata (" +
    "  `address` STRING, " +
    "  `symbol` STRING " +
    ") WITH (" +
    "  'connector' = 'starrocks', " +
    "  'jdbc-url' = '" + STARROCKS_JDBC_URL + "', " +
    "  'scan-url' = '" + STARROCKS_LOAD_URL + "', " +
    "  'database-name' = '" + STARROCKS_DATABASE + "', " +
    "  'table-name' = 'token_metadata', " +
    "  'username' = '" + STARROCKS_USERNAME + "', " +
    "  'password' = '" + STARROCKS_PASSWORD + "'" +
    ")"
);
DataStream<TokenData> historicalSymbols = tEnv.toDataStream(
    tEnv.from("starrocks_token_metadata"), TokenData.class
);

This stream is connected into both enrichment stages via KeyedCoProcessFunction. The processElement2 handler pre-populates the RocksDB cache for any token address we have seen before, so by the time live events arrive, most lookups are already warm.

Stage 1: Enrich the fromToken

We partition by fromToken and connect the metadata stream partitioned by address. The enricher checks RocksDB first; only on a cache miss does it call the external API, and then immediately saves the result:

DataStream<SwapWithFromSymbol> streamWithFromSymbol = streamWithSolPrice
    .keyBy(value -> value.exchange.fromToken != null
        ? value.exchange.fromToken : UNKNOWN_TOKEN_ADDRESS)
    .connect(historicalSymbols.keyBy(s -> s.address != null
        ? s.address : UNKNOWN_TOKEN_ADDRESS))
    .process(new EnrichFromSymbolFunction())
    .name("EnrichFromSymbol");

The result is a SwapWithFromSymbol -- a wrapper that carries both the original SwapWithPrice and the resolved fromSymbol string.

The network reshuffle

Once Stage 1 emits SwapWithFromSymbol, we can now keyBy on toToken. This triggers a network shuffle: Flink redistributes the stream so all records with the same toToken land on the same task slot. Yes, this is a network hop, but it is a single one, and it is unavoidable when you need coherent keyed state for two different keys. The alternative -- hitting a remote service for every record -- is multiple network hops per record with no caching benefit.

Stage 2: Enrich the toToken

DataStream<OutputSwapData> enrichedStream = streamWithFromSymbol
    .keyBy(value -> value.swap.exchange.toToken != null
        ? value.swap.exchange.toToken : UNKNOWN_TOKEN_ADDRESS)
    .connect(historicalSymbols.keyBy(s -> s.address != null
        ? s.address : UNKNOWN_TOKEN_ADDRESS))
    .process(new EnrichToSymbolFunction())
    .name("EnrichToSymbol");

The same RocksDB lookup pattern applies. Once both enrichments are done, we have a fully resolved OutputSwapData with the token symbols and USD volume.

The shared enricher logic

Both enrichment functions share a base class to avoid duplicating the API call and state management code:

public static class BaseSymbolEnricher implements Serializable {
    protected transient ValueState<String> symbolState;
    private transient HttpClient httpClient;

    protected String fetchSymbolFromApi(String address) {
        if (SOL_ADDRESS.equals(address)) return "SOL";
        // HTTP call to token metadata API, returns symbol or "UNKNOWN"
    }
}

Note the transient keyword on the HTTP client and state handle. Flink serializes operator state across the cluster; Java's HttpClient is not serializable. Declaring it transient means Flink will skip it during serialization, and the open() method re-initializes it when the task starts on each worker.

3. USD volume and token price calculation

Once the enriched record is assembled, we calculate two derived metrics.

USD volume is the dollar value of the swap. The logic checks which side of the trade is a stable or known-price asset:

private double calculateUsdAmount(ExchangeData e, double solPrice) {
    if (solPrice <= 0) return 0.0;
    if (USDC_ADDRESS.equals(e.fromToken) || USDT_ADDRESS.equals(e.fromToken))
        return e.fromAmount;
    if (USDC_ADDRESS.equals(e.toToken) || USDT_ADDRESS.equals(e.toToken))
        return e.toAmount;
    if (SOL_ADDRESS.equals(e.fromToken)) return e.fromAmount * solPrice;
    if (SOL_ADDRESS.equals(e.toToken))   return e.toAmount * solPrice;
    return 0.0;
}

Token price is the per-token dollar price of the non-quote asset in the swap:

private double calculateTokenPrice(ExchangeData e, double usdVolume, double solPrice) {
    boolean fromIsQuote = SOL_ADDRESS.equals(e.fromToken)
        || USDC_ADDRESS.equals(e.fromToken)
        || USDT_ADDRESS.equals(e.fromToken);
    boolean toIsQuote = SOL_ADDRESS.equals(e.toToken)
        || USDC_ADDRESS.equals(e.toToken)
        || USDT_ADDRESS.equals(e.toToken);
    if (!fromIsQuote && e.fromAmount > 0) return usdVolume / e.fromAmount;
    if (!toIsQuote   && e.toAmount > 0)   return usdVolume / e.toAmount;
    if (SOL_ADDRESS.equals(e.fromToken) || SOL_ADDRESS.equals(e.toToken)) return solPrice;
    return 0.0;
}

These two fields make downstream aggregations simple. A SUM(usdVolume) in StarRocks gives you 24h volume without any joins.

What we learned

The operational lesson here is bigger than any single Flink API. When you are building high-throughput pipelines, the instinct to reach for Redis, Memcached, or an external service for every enrichment step is a trap. Those tools have their place, but they add network hops, connection pools, and operational overhead to every single record you process.

Flink's stateful primitives exist to eliminate those hops. The patterns we landed on:

Broadcast state for volatile, shared data. If a value changes frequently but every worker needs the latest version -- a price, a config flag, a threshold -- broadcast it. The source can be the stream itself. Don't poll an external service.

Sequential keyed joins for multi-dimensional enrichment. If you need to join on two different keys, pipeline the enrichments. Each stage handles one key with coherent local state. The network shuffle between stages is the cost of correctness, and it is a fixed cost per record rather than a variable external call.

RocksDB for unbounded state. If your key space grows without bound (new tokens, new users, new whatever), heap state will eventually fail. RocksDB with incremental checkpointing gives you disk-backed state that survives job restarts without the overhead of a full snapshot on every checkpoint.

Parameterize your topology. Decouple business logic from infrastructure choices at the operator level. The enrichment code has no idea whether its output goes to Kafka or StarRocks -- that is a deployment-time decision, not a code-time one.

Running this in production for several months, our Kafka consumer lag stays flat, external API calls dropped by over 95%, and the pipeline handles traffic spikes without manual intervention. The whole thing runs as a single Flink job with no Redis cluster, no dedicated metadata service, and no external dependencies beyond Kafka and StarRocks.
